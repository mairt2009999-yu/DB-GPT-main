一、总体架构：三层鉴权 + 一处任务建模

上游网关 / BFF 认证已完成 · 注入身份头：X-User-Id / Authorization
DB-GPT FastAPI
L1 身份解析
把请求头解析为可信的用户身份（Principal）
读取网关注入的 X-User-Id。
roles 不在请求头里——需要判断权限时，L1 按需调用外部 User 服务拉取该用户的角色列表，避免每次请求都拉取。
拉取结果缓存在本地（短 TTL），减少对外部服务的依赖。
DB-GPT 管理员不是请求头标记，而是通过“现有系统中的特定角色编码”映射得出。
输出：Principal { user_id, roles, is_admin }
缺少必填头 → 401 未授权
L2 资源隔离
控制用户能看到哪些 DB-GPT 内部资源
资源包括：GptsApp（AI 应用）、数据源、会话、Prompt 模板、Flow 等。
规则：普通用户只能看自己创建的资源；DB-GPT 管理员可看全部资源；已发布的 App 对所有用户可见。
实现方式：所有列表/查询接口自动追加 WHERE owner=me；编辑/删除前验证是否为创建者。
越权访问 → 403 禁止
L3 行级 RLS
控制用户在目标数据库里能看到哪些行
LLM 生成 SQL 后，系统强制拦截，在 SQL 真正发给数据库之前：
① 解析 SQL 里涉及的所有表；
② 向上游权限服务查询「当前用户 × 每张表」的可见范围（返回 WHERE 条件片段）；
③ 把条件片段注入 SQL 对应位置（AST 改写，非字符串拼接）；
④ 执行改写后的 SQL。
用户感知不到这个过程，LLM 不参与，任何路径无法绕过。
无权限或服务不可用 → fail-close 拒绝执行
任务建模 审计记录
记录每次分析的执行过程，支持合规审计和问题复盘
每次用户发起数据分析，在 gpts_conversations 表写入：
· task_status：执行状态（等待中 / 运行中 / 成功 / 失败 / 已取消）
· executed_sql：实际打到数据库的 SQL（含权限改写后的版本）
· rls_snapshot：执行时刻使用的权限谓词快照（便于事后解释“当时为什么能/不能看到某条数据”）
这些字段仅供审计，不影响用户功能。

RLSAwareSQLExecutor.execute()
后端拦截 · SQL 权限强制改写 LLM 生成的 SQL 到达这里后，不会直接发给数据库。
系统会先查出当前用户对每张表的可见范围（例如“只能看华南区数据”），
再把这个限制条件强制拼进 SQL，最后才执行。
这是一个后台自动过程，用户不感知、LLM 不参与、任何人无法绕过。
① sqlglot 解析 SQL → 收集涉及的所有表 覆盖 JOIN / 子查询 / CTE / UNION / 自连接 新增

关键变化总览：
① 入口处先解析身份 Principal（取代原来的 mock user）；
② 资源加载（Skill / 数据源）按 owner 过滤；管理员角色可放宽跨 owner 查看；
③ Prompt / ReAct 改造：把用户身份与可见表/列写入 system prompt，仅用于提升 SQL 通过率，不是安全底线；
④ 模型生成 SQL 后，统一进入 RLSAwareSQLExecutor：调上游谓词接口 + sqlglot AST 改写，不依赖 LLM 自觉调用（LLM 可能被 prompt 注入或偷懒跳过）；
⑤ 执行后写入 executed_sql / rls_snapshot 审计快照，便于事后复盘。

为什么不让 Agent 自己调谓词接口（ReAct 工具调用方式）？
· LLM 可能被 prompt 注入，跳过工具直接返回原 SQL；
· LLM 自己拼谓词到 SQL 里，对 LEFT JOIN / 聚合 / CTE 边界处理不可靠；
· 安全决策必须发生在不可被 LLM 绕过的链路上 → 放在后端 Executor 层。

三、资源权限设计细节（L2 资源隔离）

这一层解决的不是“数据库里哪些数据行可见”，而是“用户能不能看到/修改 DB-GPT 自己的资源”。
例如：能不能看到某个 GptsApp、能不能使用某个数据源、能不能打开别人的会话记录。
初版采用 owner + 管理员角色放宽 的简单模型，不做完整 RBAC；管理员身份来自现有系统中的特定角色编码。

1. 参与判定的身份信息

  • user_id：当前登录用户 ID
  • roles：不在请求头里；需要时调用 User 服务获取并缓存
  • is_admin：由现有系统中的特定角色编码映射得到，用于放宽“跨 owner 查看”的限制

2. 当前纳入控制的资源

  • GptsApp：AI 应用 / 助手
  • Datasource：数据源连接配置
  • Conversation：会话与任务记录
  • Prompt / Flow：提示词模板、工作流配置
  • File / dbgpts：文件、插件等资源

3. 设计原则

  • 同用户归属：普通用户默认只能看自己创建的资源
  • 管理员放宽：拥有指定角色编码的管理员可看全部资源
  • 公开例外：仅 GptsApp 支持 published，对所有用户可见
  • 接口收口：不靠前端隐藏按钮，后端接口强制校验

资源权限判定矩阵

① 资源 owner = 当前用户；或
② GptsApp 且 published = true。

否则返回 403。

用于管理员代运维、资源纠偏、异常清理。

无管理员角色 → 403 Forbidden

后端实现方式（不依赖前端）

请求进入接口
→ L1 先得到 Principal（user_id / roles）
→ L2 在 DAO / Service 层自动追加 owner 过滤条件
→ 查询结果返回给业务层
→ 若是编辑 / 删除动作，则额外执行 assert_owner 断言
→ 不满足条件时直接返回 403，不依赖前端“是否隐藏按钮”
